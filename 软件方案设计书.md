## 一、项目概述

### 1.1 项目目标

开发一套通用视觉检测平台，面向半导体芯片检测行业，提供以下核心能力：

- 图像处理算子库（基于 Halcon 封装）
- 多协议通讯库（串口、Modbus、PLC、TCP/UDP、WebAPI、SECS/GEM、GPIB 等）
- 设备管理（相机、光源、运动控制卡）
- 可配置的检测流程编排
- 系统操作库（日志、文件、数据库、报警）

### 1.2 核心设计理念

**不做万能平台，做可扩展框架。** 接口层面预留完整的扩展空间，实现层面按需迭代，先让一个客户用起来，再逐步丰富。

### 1.3 开发规模

| 指标 | 数值 |
|------|------|
| 建议团队规模 | 5-6 人 |
| 单人+AI 辅助周期 | 约 6 个月（MVP V1.0） |
| 完整团队周期 | 12-18 个月（V1.0 完整版） |

### 3.2 ImageLib 设计要点

**Halcon 隔离策略**：Halcon 的 `HObject`/`HImage`/`HRegion` 不直接暴露给算子，所有数据通过 `ImageData` 和 `RegionData` 封装。Halcon 版本升级时，仅需修改 `HalconBridge` 和 `ImageData` 内部实现。

**算子发现机制**：通过 `[Processor]` 特性 + 反射自动发现所有算子，工具箱自动生成。

```csharp
[Processor("Blob分析", Category = "检测", 
    Description = "通过阈值分割和连通域分析检测缺陷区域")]
public class BlobAnalysisProcessor : IImageProcessor
{
    public string Id => "BlobAnalysis";
    public string Name => "Blob分析";
    public string Category => "检测";
    
    public IReadOnlyList<ParamDef> InputParams => new[]
    {
        new ParamDef("MinArea", ParamType.Double, 100, "最小面积"),
        new ParamDef("MaxArea", ParamType.Double, 999999, "最大面积"),
        new ParamDef("Threshold", ParamType.Double, 128, "分割阈值"),
    };
    
    public async Task<ProcessResult> ExecuteAsync(ProcessContext ctx, CancellationToken ct)
    {
        // 实现
    }
}
```

### 3.5 流程引擎设计

**流程编排方式**：采用**树形大纲 + 属性面板**方案，而非拖拽流程图。

- 开发周期：约 2-3 周
- 用户体验：学习成本低，所见即所得
- 序列化：纯 JSON，易于版本管理
- 调试：支持单步执行

**流程定义模型**：

```csharp
// 顶层：检测配方
public class InspectionRecipe
{
    public string Id { get; set; }
    public string Name { get; set; }
    public List<InspectionStep> Steps { get; set; } = new();
}

// 步骤（每个步骤有明确的语义：采集/定位/检测/测量/通讯）
public class InspectionStep
{
    public string Id { get; set; }
    public string Name { get; set; }
    public StepType Type { get; set; }
    public List<ProcessorNode> SubSteps { get; set; } = new();  // 子步骤链
    public Dictionary<string, object> Config { get; set; } = new();
    public bool StopOnFailure { get; set; } = true;
}

// 最细粒度：算子节点
public class ProcessorNode
{
    public string Id { get; set; }
    public string ProcessorType { get; set; }
    public Dictionary<string, object> Parameters { get; set; } = new();
    public string InputReference { get; set; }  // 引用前面步骤的输出
}

// 步骤类型
public enum StepType
{
    ImageAcquisition,   // 图像采集
    Alignment,          // 定位/对位
    Inspection,         // 检测
    Measurement,        // 测量
    Communication,      // 通讯
    SystemOperation,    // 系统操作
}
```

**执行模型**：顺序执行，上一步输出自动作为下一步输入，支持条件跳过。

```csharp
public class RecipeExecutor : IRecipeExecutor
{
    public async Task<RecipeResult> ExecuteAsync(InspectionRecipe recipe, CancellationToken ct)
    {
        var context = new RecipeContext();
        foreach (var step in recipe.Steps)
        {
            ct.ThrowIfCancellationRequested();
            var result = await ExecuteStepAsync(step, context, ct);
            context.SetOutput(step.Id, result);
            if (result.IsFailed && step.StopOnFailure) break;
        }
        return context.BuildResult();
    }
}
```

**扩展路径**：

```
V1.0 → 树形大纲 + 顺序执行（当前方案）
V1.5 → 增加条件分支（某步骤结果决定跳转）
V2.0 → 增加并行步骤（多步骤同时执行）
V3.0 → 升级为拖拽流程图（如果客户需要）
```

## 六、质量保证

### 6.1 测试策略

```
测试金字塔（自下而上）

          ┌─────┐
          │ E2E  │  ← 完整检测流程，客户场景模拟（10%）
        ┌─┴─────┴─┐
        │ 集成测试  │  ← 采集→处理→输出，通讯→设备（20%）
      ┌─┴─────────┴─┐
      │   单元测试    │  ← 算子、适配器、引擎、工具类（70%）
      └─────────────┘
```

### 6.2 单元测试规范

- **框架**：xUnit + Moq + FluentAssertions
- **命名**：`方法名_场景_预期结果`
- **结构**：AAA 模式（Arrange / Act / Assert）
- **必须覆盖**：正常路径、边界值、null 输入、异常路径、并发安全、资源释放
- **覆盖率目标**：ImageLib 70%+，WorkflowEngine 80%+，CommLib.Core 60%+，WPF App 30%+
- **CI 集成**：每次提交自动运行，覆盖率不达标则构建失败

### 6.3 稳定性保障

- **分层崩溃隔离**：UI 层崩溃不影响检测流程，通讯层断线自动重连，单帧异常跳过不中断
- **全局异常处理**：`DispatcherUnhandledException` + `TaskScheduler.UnobservedTaskException`
- **内存管理**：图像对象池（`Microsoft.Extensions.ObjectPool`），减少 GC 压力
- **健康检查**：关键线程心跳监控，超时自动重启
- **结构化日志**：NLog 输出，每次检测记录算子名、耗时、结果

### 6.4 版本管理

- **分支策略**：Git Flow 简化版（main / develop / feature/* / hotfix/* / release/*）
- **版本号**：语义化版本 `MAJOR.MINOR.PATCH`
- **流程文件兼容**：JSON 包含版本号，反序列化时执行版本迁移
- **维护周期**：每个版本活跃维护 6 个月，安全维护 12 个月

## 八、风险管理

| 风险 | 等级 | 应对措施 |
|------|------|---------|
| Halcon 大版本升级 API 变化 | 高 | HalconBridge 隔离，算子只操作抽象类型 |
| SECS/GEM 自研失败 | 高 | 直接采购商业库，10-20 万预算 |
| 流程编辑器性能瓶颈 | 中 | TreeView 模式，避免 Canvas 大量节点 |
| 多设备时序同步 | 中 | 硬件触发信号为主，软件辅助调度 |
| 内存泄漏（7×24 运行） | 中 | 对象池 + dotMemory 定期排查 |
| 接口设计不合理导致后期重构 | 中 | 第一周冻结接口，每接口至少一个实现验证 |
| 单人开发精力分散 | 中 | 严格按迭代计划，禁止超出范围开发 |

## 九、关键原则

1. **接口多留，实现少做** — 接口定义不花钱，实现只做当下需要的
2. **先跑通，再优化** — 3 个算子串起来能跑，比 50 个算子只能看强
3. **能用现成的绝不自己写** — 开源库、商业库、第三方控件，组合而非重造
4. **不做没有硬件的适配器** — 没实物不写代码，写了也没法验证
5. **每周末必须有可运行版本** — 不能跑的就是废品
6. **接口冻结后只增不减** — 改接口等于所有实现全部重写
7. **不做万能平台** — 第一个客户需要什么就做什么，按需扩展

### 11.1 AI 能做什么、不能做什么

## 十四、务实的注意事项

### 14.1 不要过度设计

- 接口只分三层：IImageProcessor、ICommunicationAdapter、IDevice。不要搞俄罗斯套娃式继承
- 不要一开始就做对象池、无锁队列、SIMD 优化 — 先跑通再优化
- 不要写"以后可能会用到"的代码 — 只写当前需要的
- 不要从零写 WPF 自定义控件 — 用 HandyControl、AvalonDock 等现成的

### 14.2 硬件依赖管理

- **没有硬件就不写适配器代码** — 空实现无法验证，等于白写
- 提前采购至少一套完整硬件：Basler 相机 + 三菱 PLC + 串口光源 + Modbus 设备
- 所有适配器必须在真实硬件上测试通过后才能合并

### 14.3 接口设计验证法则

**每定义一个接口，必须至少写一个实现来验证接口是否合理。** 空接口定义是纸上谈兵。

```
IImageProcessor → 至少实现 BlobAnalysisProcessor
ICommunicationAdapter → 至少实现 SerialPortAdapter
ICamera → 至少实现 BaslerCamera
IRecipeExecutor → 必须实现 RecipeExecutor
```

### 14.4 代码风格一致性

- AI 生成的代码风格可能不一致，每次生成后必须检查
- 配置 .editorconfig 强制统一缩进、命名、空格
- 关键代码手动调整后再提交，不要"差不多就行"

### 14.5 避免的常见陷阱

| 陷阱 | 为什么是陷阱 | 如何避免 |
|------|-------------|---------|
| 追求完美的流程设计器 | 用户根本不在乎连线的美观度 | 先做功能，体验优化放后面 |
| 一口气写完整文档 | 接口还在改，文档已过时 | 冻结后再写文档 |
| 全部通讯协议自研 | 开源库比你写的好 | 封装开源库，不自研协议栈 |
| 过早做性能优化 | 优化了不是瓶颈的地方 | 先跑通，用 Profiler 对准瓶颈 |
| 想一次支持所有设备 | 80% 适配器没人用 | V1.0 只做 Basler + 三菱 PLC |
| 在多个 AI 工具间频繁切换 | 每个 AI 都要重新加载上下文 | 一个阶段用一个主力工具 |
| 不写阶段记录 | 中断两周回来什么都不记得 | 每周五必须写阶段总结 |

### 14.6 如果项目卡住了

```
判断标准：一个问题卡住超过 2 小时 → 不是技术问题，是方案问题。

处理流程：
1. 停止死磕
2. 把问题写在"待解决问题.md"中
3. 问自己：有没有更简单的方案？
4. 如果有 → 用简单方案，继续推进
5. 如果没有 → 标记为"阻塞"，先做其他模块
```
