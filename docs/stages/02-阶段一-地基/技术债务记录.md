## 二、债务详情

### DEBT-001：Dictionary<string, object> 反序列化类型问题

**模块**：WorkflowEngine.Core
**类型**：代码质量
**严重程度**：低
**优先级**：P3
**计划版本**：V0.2.0

#### 问题描述
`ProcessorNode.Parameters` 是 `Dictionary<string, object>` 类型，使用 System.Text.Json 反序列化后，值的类型是 `JsonElement` 而不是原始的 CLR 类型（int, double, string, bool）。

#### 当前状态
功能正常，算子中通过 `context.GetParam<T>()` 方法做类型转换（使用 `Convert.ToDouble` 等），不影响使用。

#### 重构方案
实现自定义 `JsonConverter<Dictionary<string, object>>`，在反序列化时将 JsonElement 转换为对应 CLR 类型：
- JsonValueKind.Number → 优先 int，超出范围转 double
- JsonValueKind.String → string
- JsonValueKind.True/False → bool
- JsonValueKind.Null → null
- JsonValueKind.Object → Dictionary<string, object>
- JsonValueKind.Array → List<object>

#### 收益
- 代码更优雅，减少类型转换代码
- 避免潜在的类型转换异常
- 参数使用体验更好

**模块**：所有测试项目
**类型**：依赖
**严重程度**：低
**优先级**：P3
**计划版本**：V0.2.0

#### 问题描述
Moq 4.20.0 版本存在已知的低严重性安全漏洞（GHSA-6r78-m64m-qwcf）。

#### 当前状态
- 漏洞等级：低
- 仅影响测试项目，不影响生产代码
- 功能正常使用

#### 重构方案
升级 Moq 到最新稳定版（4.20.70+ 或更高版本）。

#### 收益
- 消除安全漏洞警告
- 获得最新的 Bug 修复和功能改进

### DEBT-003：缺少代码覆盖率统计工具

**模块**：测试基础设施
**类型**：工具链
**严重程度**：中
**优先级**：P2
**计划版本**：V0.2.0

#### 问题描述
目前没有代码覆盖率统计工具，无法精确衡量测试覆盖情况，只能靠估算。

#### 当前状态
- 覆盖率靠人工估算
- 无法识别未覆盖的代码分支
- 无法设置 CI 门禁

#### 重构方案
引入 coverlet.msbuild + ReportGenerator：
1. 测试项目添加 coverlet.msbuild NuGet 包
2. 运行测试时收集覆盖率：`dotnet test /p:CollectCoverage=true`
3. 生成 HTML 报告：ReportGenerator
4. 设置覆盖率目标门禁

#### 收益
- 精确的覆盖率数据
- 识别测试盲区
- 支持 CI 质量门禁

### DEBT-004：无 CI/CD 流水线

**模块**：基础设施
**类型**：工具链
**严重程度**：中
**优先级**：P2
**计划版本**：V0.2.0

#### 问题描述
目前没有自动化的 CI/CD 流水线，代码提交后需要手动构建和测试。

#### 当前状态
- 本地手动运行 `dotnet build` 和 `dotnet test`
- 无自动化质量门禁
- 无自动化发布流程

#### 重构方案
使用 GitHub Actions（或 Azure DevOps）搭建 CI/CD：
1. PR 触发：编译 + 单元测试 + 覆盖率检查
2. 主分支合并：编译 + 全量测试 + 打包发布
3. 标签推送：生成 Release 包

#### 收益
- 每次提交自动验证质量
- 减少人工操作失误
- 提升发布效率

### DEBT-006：通讯协议未做真实设备联调

**模块**：CommLib
**类型**：测试
**严重程度**：中
**优先级**：P2
**计划版本**：V0.2.0

#### 问题描述
串口和 Modbus TCP 适配器只做了单元测试（Mock 或模拟），未与真实设备联调。

#### 当前状态
- 串口：未连接真实串口设备测试
- Modbus TCP：未连接真实 PLC 测试
- 协议解析逻辑未经过真实数据验证

#### 重构方案
1. 使用串口调试助手 + 虚拟串口测试 SerialPortAdapter

3. 如有条件，连接真实 PLC 进行联调
4. 补充协议一致性测试

#### 收益
- 确保通讯协议实现正确性
- 发现边界情况和异常场景
- 验证连接稳定性
