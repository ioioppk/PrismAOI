## 二、技术决策

### 决策 1：Halcon 隔离策略
**内容**：通过 ImageData/RegionData 封装类完全隔离 Halcon 的 HObject/HRegion，外部代码不直接引用 Halcon 类型。
**理由**：Halcon 版本升级时只需修改 HalconBridge 层，不影响上层算子和业务逻辑。
**替代方案**：直接暴露 HObject，开发效率更高但耦合严重。

### 决策 2：Modbus TCP 原生实现
**内容**：ModbusTcpAdapter 采用原生 TcpClient + 手动组包实现，不依赖 NModbus4。
**理由**：NModbus4 是 .NET Framework 时代的库，在 .NET 6 下通过兼容模式运行，存在兼容性风险。原生实现代码量可控（~400行），且完全可控。
**替代方案**：继续使用 NModbus4，开发更快但兼容性存疑。

### 决策 3：TreeView 流程编辑器
**内容**：采用 TreeView + 属性面板的方式编排检测流程，而非拖拽式流程图。
**理由**：开发周期短（2-3周 vs 2-3个月），学习成本低，JSON 序列化简单。
**替代方案**：拖拽流程图，用户体验更好但开发成本极高。

### 决策 4：对象池设计
**内容**：ImagePool 采用 ConcurrentBag 实现简单对象池，用于复用 ImageData 对象。
**理由**：7×24 运行场景下减少 GC 压力，避免内存碎片。
**替代方案**：Microsoft.Extensions.ObjectPool，功能更全但引入额外依赖。

### 决策 5：枚举序列化
**内容**：RecipeSerializer 使用 JsonStringEnumConverter 实现枚举的字符串序列化/反序列化。
**理由**：JSON 可读性更好，与 demo_recipe.json 格式一致，方便人工编辑。
**替代方案**：默认数值序列化，体积略小但可读性差。

## 三、遇到的问题

| 问题 | 原因 | 解决方案 | 是否解决 |
|------|------|---------|---------|
| NModbus4 在 .NET 6 下兼容性问题 | NModbus4 基于 .NET Framework 构建 | 改为原生 TcpClient 实现 Modbus TCP 协议 | ✅ |
| Interlocked.Increment 不支持 ushort | C# 泛型限制 | 将 _transactionId 改为 int 类型 | ✅ |
| Halcon 未安装时的编译问题 | HalconDotNet.dll 不存在 | 使用 HAS_HALCON 编译常量 + 条件编译 | ✅ |
